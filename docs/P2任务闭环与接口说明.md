# campus-system-p2：P2任务、闭环逻辑、接口与示例

本文用于说明 P2 教师端需要完成什么、哪些逻辑必须在 P2 内部闭环、当前完成度如何、与 P1/P3/前端协作的接口是否有效，以及如何用示例输入看到最终输出效果。

## 1. P2定位

P2 是教师端考试分析子系统，不负责 PDF 切题，也不负责真实题库检索本身。

P2 的核心职责是：

1. 读取 P1 输出的规范化试卷 JSON。
2. 读取阅卷后的班级成绩数据。
3. 将“题目、得分、知识点”对齐。
4. 生成教师可读的考试诊断、薄弱知识点分析和讲评建议。
5. 向 P3 生成标准化的题库检索请求，供后续“举一反三”和学生强化训练使用。
6. 向 GUI/API/CLI 提供统一的分析结果和报告导出能力。

换句话说，P2 是“考试数据进入教师端业务分析”的中间层：上游吃 P1 的结构化题目，下游给 P3 提供可检索的薄弱知识点需求。

## 2. P2需要完成的任务

### 2.1 输入处理

- 接收 P1 的 `paper.v0.1` 规范化试卷 JSON。
- 接收阅卷成绩文件，当前支持 `.xlsx`、`.xlsm`、`.csv`、`.txt`。
- 校验题目字段是否满足 P2 分析需要。
- 校验成绩文件是否包含题号、满分、平均分或得分率。
- 保留小题格式，例如 `17(1)`、`17(2)`，避免合并丢失信息。

### 2.2 试题分析

- 按题号匹配试卷题目和成绩记录。
- 计算每题平均分、满分、得分率、失分率。
- 按得分率生成风险等级：
  - `critical`：严重薄弱。
  - `warning`：需要关注。
  - `normal`：基本正常。
- 保留原题的题型、难度、答案、解析、知识点候选。

### 2.3 知识点确认与聚合

- 读取 P1 或大模型给出的 `knowledge_candidates`。
- 生成 P2 的 `confirmed_knowledge_points`。
- 按知识点聚合相关题目。
- 计算每个知识点的综合得分率和风险等级。
- 生成教师端讲评建议。

### 2.4 教师讲评报告

- 生成本次考试概览。
- 生成重点讲评题目列表。
- 生成薄弱知识点列表。
- 生成可操作的课堂讲评建议。
- 导出 JSON、Markdown、Word 三类结果。

### 2.5 P3协作请求

- 根据薄弱知识点生成 `p3_search_requests`。
- 每条请求包含知识点编码、题型、难度范围、数量限制、需排除的原题 ID。
- P2 不直接承担真实题库检索，但输出已经可以被 P3 接收。

### 2.6 前端/API

- 提供 FastAPI 后端接口。
- 提供 React/Vite 前端页面。
- 前端可上传 P1 JSON 和成绩文件，查看分析结果、知识点诊断和报告导出；P3 原始请求保留在后端输出中，不作为用户界面内容展示。

## 3. P2内部必须闭环完成的逻辑

下面这些逻辑必须由 P2 自己完成，不能依赖 P1/P3 补齐。

| 闭环逻辑 | 当前实现 | 说明 |
|---|---|---|
| P1 JSON 读取与校验 | 已完成 | `campus_p2_core/p1_input/normalized_paper.py` |
| 成绩文件读取 | 已完成 | `campus_p2_core/p2_teacher/score_loader.py` |
| 题号匹配 | 已完成 | 支持普通题号和 `17(1)` 这类小题 |
| 得分率/失分率计算 | 已完成 | 由满分、平均分或得分率推导 |
| 风险等级判断 | 已完成 | 输出 `critical`、`warning`、`normal` |
| 知识点确认 | 已完成 | 第一阶段直接采用 P1 候选知识点，保留置信度和来源 |
| 薄弱知识点聚合 | 已完成 | 同一知识点跨题汇总 |
| 教师建议生成 | 已完成 | 当前为规则化建议，后续可接大模型润色 |
| P3 检索请求生成 | 已完成 | 输出标准 JSON 请求数组 |
| 报告导出 | 已完成 | JSON、Markdown、DOCX |
| API 服务 | 已完成 | `backend/app/main.py` |
| 前端展示 | 已完成 | `frontend/src/main.tsx`、`frontend/src/styles.css` |

第一阶段中，“知识点确认”和“教师建议生成”采用轻量规则闭环，这是合理的 Demo 策略：先保证数据链路稳定，后续再替换为更强的大模型服务或人工确认流程。

## 5. 与其他人协作的接口

### 5.1 P1 -> P2：规范化试卷

P1 给 P2 的输入是 JSON 文件，示例路径：

```text
examples/normalized_paper_demo.json
```

最小有效结构：

```json
{
  "schema_version": "paper.v0.1",
  "paper_id": "paper_demo_20260704_001",
  "subject": "math",
  "stage": "senior_high",
  "grade": "高二",
  "questions": [
    {
      "question_id": "paper_demo_20260704_001_q001",
      "question_no": "1",
      "question_type": "single_choice",
      "stem_text": "已知集合 A={x|x>1}，B={x|x<5}，求 A∩B。",
      "answer": "A",
      "solution": "同时满足 x>1 与 x<5，故交集为 (1,5)。",
      "knowledge_candidates": [
        {
          "code": "KP-FOUND-SET-001",
          "name": "集合的表示与运算",
          "confidence": 0.9,
          "source": "demo"
        }
      ],
      "difficulty": 1,
      "parse_confidence": 0.98,
      "needs_review": false
    }
  ]
}
```

接口有效性判断：

- 有效。P2 已能读取并校验该结构。
- `question_no` 是 P2 与成绩文件匹配的关键字段，P1 必须稳定输出。
- `knowledge_candidates` 是 P2 生成知识点诊断和 P3 请求的关键字段，建议 P1 或大模型侧必须尽量给出 `code` 和 `name`。
- 若真实 P1 输出中存在题号识别错误，P2 会产生 `warnings`，但不应在 P2 内部修 OCR 错误。

### 5.2 阅卷数据 -> P2：成绩文件

示例路径：

examples/sample_exam_scores.xlsx

接口有效性判断：

- 当前最适合 Demo 的输入是“按题汇总的班级平均分”。
- 后续如要做学生个人报告，可扩展为“学生-题号矩阵”，但这属于 P2/P3 下一阶段增强，不影响当前教师端 Demo。

### 5.3 P2 -> P3：举一反三检索请求

P2 输出字段：

```json
{
  "p3_search_requests": [
    {
      "knowledge_point_codes": ["KP-FUNC-DERIV-002"],
      "question_type": null,
      "difficulty_range": [1, 4],
      "limit": 5,
      "exclude_question_ids": ["paper_demo_20260704_001_q021"]
    }
  ]
}
```

接口有效性判断：

- 有效。P2 已经能稳定输出该数组。
- P3 只需要按 `knowledge_point_codes`、`difficulty_range`、`limit` 检索题库，并排除 `exclude_question_ids`。
- `question_type` 当前允许为 `null`，表示题型不限；P3 可以先忽略该字段。
- 后续如果 P3 有固定返回格式，P2 前端可增加“推荐练习题”区域展示 P3 返回结果。

### 5.4 P2 -> 前端/API

后端接口：

```text
GET  /api/health
GET  /api/model/status
GET  /api/p2/demo
GET  /api/p2/examples/paper
GET  /api/p2/examples/scores
POST /api/p2/analyze
POST /api/p2/reports/docx
POST /api/p2/reports/markdown
```

`POST /api/p2/analyze` 接收：

| 字段 | 类型 | 说明 |
|---|---|---|
| `paper_file` | file | P1 规范化试卷 JSON |
| `score_file` | file | 阅卷成绩文件 |
| `exam_id` | text | 考试 ID |
| `class_name` | text | 班级名 |

接口有效性判断：

- 有效。前端可直接上传示例文件并查看分析结果。
- CLI、API、GUI 共用 `campus_p2_core`，符合“GUI 不包含算法逻辑”的会议要求。

## 8. 后续对接建议

1. P1 同学提供真实切题后的 `paper.v0.1` 文件后，优先校验 `question_no`、`question_id`、`knowledge_candidates` 三个字段。
2. P3 同学先按 `knowledge_point_codes` 接收 P2 请求，返回题目列表即可，不必第一版就做复杂排序。
3. 教师端第一版重点展示“题目诊断、知识点薄弱、讲评建议、举一反三请求”，不要在 7.31 前扩散到学生长期画像。
4. 大模型第一阶段只作为知识点标注或报告润色增强，不应阻塞 P2 主链路。
5. 真实学校数据进入后，要重点观察题号格式、小题编号和知识点编码是否稳定。
